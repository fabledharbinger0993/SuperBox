Core Pipeline — each tool feeds the next

Library Audit — Database & Physical Scan
Two simultaneous passes: a database audit (track counts, tag coverage, missing paths, dead drives) and a physical library scan (every file on disk, size, format breakdown, orphans, trash-adjacent files). Read-only — safe to run with RekordBox open.
Read-Only
What does Audit actually check?

It opens the database in read-only mode and runs three passes:

  • Snapshot — how many tracks, playlists, cue points, and how many are missing BPM, key, artist, album, or genre tags.
  • Path check — every file path stored in the database is verified to still exist on disk. Missing files show up here.
  • Orphan scan — files on disk that RekordBox doesn't know about yet.

Nothing is written. You get a clean report and a baseline to compare against after you're done.

Can anything go wrong?

No. Audit is the one command that cannot hurt you. It only reads. The only thing that can go wrong is if the database path in config.py is wrong — in that case it will error with a clear message before touching anything.

⚠ Disconnected drive detected
These dead roots have been pre-filled in Relocate — Fix Broken Paths below. Run that step next.
Tag Tracks — Detect BPM & Musical Key
Reads each audio file and writes BPM and musical key into the file's tags. Skips files that already have tags — won't overwrite your Mix In Key work unless you ask it to.
Writes Tags
How does it detect BPM and key?

BPM is calculated by librosa beat tracking, which analyzes the actual audio waveform for beat patterns — not guessing from the filename. Result is sanity-checked (must be between 30–300 BPM).

Key uses the Krumhansl-Schmuckler algorithm on the audio's chroma features via librosa. The result is translated to whichever notation RekordBox uses (Camelot, Open Key, or standard — whatever's already in your database).

Files that already have a BPM or key tag are skipped by default. Use Force Overwrite only if you want to replace existing values.

What could go wrong?

This modifies the file tags (ID3, AIFF metadata etc.) — not the audio itself. Your audio waveform is untouched.

Edge cases to know about:

  • BPM detection can be fooled by very long intros or unusual time signatures. Review results in RekordBox afterward.
  • Key detection is an estimate — it works well on most house, techno, and disco, but can struggle with atonal or heavily processed tracks.
  • If a file can't be read (corrupt, DRM, unsupported format), it's logged as an error and skipped — it does not stop the rest of the batch.
Find Duplicate Tracks
Acoustically fingerprints every file and groups the ones that are actually the same song — even if they have different filenames, formats, or bitrates. Outputs a CSV report. Never deletes anything.
Read-Only
How does it know two files are the same song?

It uses Chromaprint (the same fingerprinting engine behind AcoustID / MusicBrainz). It analyzes the actual audio waveform — not the filename or tags — and produces a fingerprint unique to that recording.

Two files with an identical fingerprint are the same recording. This catches: same file in two formats (MP3 + AIFF), same file named differently, files with different tags but identical audio.

Within each group of duplicates, files are ranked using the RARP hierarchy to suggest which to keep:

  • PN (Pioneer Numbered) — filename starts with digits and a separator, e.g. 01 - Title. Suggests it came from a curated source.
  • MIK (Mixed In Key tagged) — has a TKEY/initialkey tag already written by Mix In Key.
  • RAW — neither of the above.

The CSV is yours to review. Nothing is deleted or moved.

How long does this take?

Chromaprint analyzes the first 120 seconds of each file. For a large library (10,000+ tracks), expect 30–90 minutes depending on your drive speed. Progress is logged every 100 files.

You can run this with RekordBox open — it never touches the database.

Prune Duplicates — Review & Remove
Load a duplicate report, review each group side-by-side, pick what to keep, and move the rest to a labeled recovery folder in Trash. Nothing is permanently deleted. RekordBox must be closed.
Writes DB + Files
How does the review work?

Each duplicate group shows the best quality copy at the top — ranked by format (AIFF/WAV over MP3), then file size, then RARP tier. Lower-quality copies are listed indented beneath it.

You get two Select All controls:

  • Select All Best — checks every top-ranked copy (the ones to keep)
  • Select All Lower Quality — checks all the duplicates below them

These two are mutually exclusive. You can also override any individual checkbox.

The ▶ Preview button opens each file in your default audio player so you can listen before deciding.

What actually happens when I prune?

Step 1 — Database: Selected tracks are removed from RekordBox's database. A timestamped backup is created before anything is touched.

Step 2 — Files: Selected audio files are moved to ~/.Trash/SuperBox_Pruned_[timestamp]/ — a labeled folder inside Trash. They are not deleted. They sit in Trash until you empty it yourself.

If you realize you made a mistake, open Trash, find the SuperBox_Pruned_ folder, and move the files back.

Why so many confirmation steps?

Because this operation touches both the database and the files, and there's no single-step undo. Each confirmation panel appears in a different position on screen so you have to move your cursor to proceed — you can't click through by accident.

  • Step 1 confirms what you're removing (count + summary)
  • Step 2 shows the exact file list for a final read-through
  • Step 3 is the live execute — requires RekordBox to be closed

Relocate — Fix Broken File Paths
When you move or rename your music folders, RekordBox loses the tracks. This updates the database to point to where the files actually are now. RekordBox must be closed.
Writes DB
How does it find files that have moved?

For each track whose old path no longer exists, it tries three strategies in order:

  • Exact — the same relative path exists under the new root.
  • Hash — SHA256 of the first 64 KB matches a file under the new root. Catches files that were renamed but have identical content.
  • Fuzzy — filename similarity ≥ 90%. Catches minor edits, encoding fixes, format changes (e.g. .aif → .aiff).

Tracks that can't be matched by any strategy are logged as warnings and left unchanged — they are not deleted or broken further.

The old root does not need to exist on disk — it's matched as a text prefix against the paths stored in the database. If you renamed a drive or moved everything to a new mount point, just enter what the old path was.

What if I type the old root wrong?

If the old root doesn't match any paths in the database, zero tracks will be processed and you'll see a warning in the output. Nothing will be changed. Run Audit first to confirm what paths the database actually contains.

⚠ RekordBox is open. Close it before relocating.
Import — Add Tracks to RekordBox
Preview first to see exactly what will be added — then import. RekordBox must be closed for the real import. A timestamped backup is created automatically before anything is written.
Writes DB
What does Preview show?

Preview walks your music folder and checks each file against what's already in the database — no writes, no changes. You'll see:

  • How many tracks are new (will be imported)
  • How many are already in the DB (will be skipped)
  • Any files it can't read or extract metadata from

Always run Preview first. If the numbers look right, hit Import.

What exactly gets written to the database?

For each new track, it creates a row in DjmdContent with the metadata extracted from the file's tags:

  • Title, artist, album, genre, year, track number
  • BPM (stored as BPM × 100 — that's how RekordBox expects it)
  • Musical key (translated to RekordBox's internal key ID)
  • Duration, bitrate, sample rate, bit depth, file path

Tracks already in the database are skipped entirely — no duplicates created, no overwriting.

The operation commits in batches of 250. If anything fails mid-batch, the entire batch rolls back automatically. You won't get half-imported tracks.

What if something goes wrong?

Before the first write, a timestamped copy of master.db is saved to ~/rekordbox-toolkit/backups/. If the import fails or something looks wrong afterward, you can restore from that file.

RekordBox must be closed — the server checks this before starting. If it's open, the import will be blocked here before anything is touched.

A single bad file will not abort the batch — errors are captured per-file and logged, and the rest of the import continues.

⚠ RekordBox is open. Close it before importing.
Link Playlists — Connect Tracks to Folders
Matches imported tracks to RekordBox playlists based on the folder they live in. If your playlists were built from your folder structure, this connects them automatically.
Writes DB
How does the matching work?

For each track, it walks up the folder path toward your music root and compares folder names against playlist names in the database:

  • Exact match first — folder name equals playlist name exactly.
  • Fuzzy match second — if no exact match, it tries similarity scoring. Threshold is 85% similarity. Only applied to names 4 characters or longer (to avoid false positives on short names).

If no match is found, the track is left unlinked and a warning is logged. No track is ever removed from a playlist it was already in.

What if my folder names don't match my playlist names?

Tracks that can't be matched are logged and left alone — they stay in the database, just not assigned to a playlist. You'll see the count in the output.

The fuzzy match handles common differences like trailing spaces, slight spelling variations, or accented characters. If the mismatch is large (e.g. completely different naming conventions), you may need to manually link those tracks in RekordBox.

⚠ RekordBox is open. Close it before linking.

Optional & Transformative — run after the core pipeline is clean

Normalize Loudness (Optional)
Adjusts every track's volume so they all hit at the same level on the CDJs. Target is −8.0 LUFS — the DJ standard. This rewrites the actual audio file.
Rewrites Audio
What exactly happens to my files?

Each file is analyzed for integrated loudness (EBU R128 standard). If it's outside −8.0 LUFS ± 0.5 LUFS, the file is re-encoded:

  • MP3 → re-encoded at 320 kbps CBR
  • AIFF/WAV → re-encoded losslessly at the same bit depth as the original

Files already within the tolerance window are left untouched.

The process: original is renamed to .bak, new file is written, soundfile validates the new file is readable, then — and only then — the .bak is deleted. If validation fails, the .bak is kept and you still have your original.

Should I normalize my files?

Only if inconsistent levels are a real problem for you. −8.0 LUFS is the correct target for DJ use — it leaves headroom for the mixer and matches what most commercial releases already land at.

Things to know before you run this:

  • This is the highest-risk operation in the toolkit. Back up your drive first.
  • MP3 re-encoding is a lossy process — even at 320 kbps, you're going through an encode cycle. For AIFF/WAV files, re-encoding is lossless.
  • Tracks that are already at the right level are skipped entirely — this will not process your whole library unnecessarily.
  • Do not run this while the drive is in use by RekordBox or during a gig.
Convert — Change Audio Format
Convert your entire library (or a folder) to a different audio format. Supports MP3, WAV, AIFF, and FLAC. Metadata is preserved.
Modifies Files
Which format should I choose?
  • MP3 — smallest files, universal compatibility, lossy compression. Best for: storage constraints, portability.
  • WAV — lossless, uncompressed. Best for: archival, mastering, when file size isn't a concern.
  • AIFF — lossless, mainly used on Mac. Best for: Mac DJs, archival.
  • FLAC — lossless, compressed to ~60% of WAV size. Best for: archival with smaller file sizes, high-fidelity listening.

RekordBox works with all formats equally. Choose based on your storage space, backup strategy, and playback device.

Will my tags be preserved?

Yes. All metadata (BPM, key, cue points, hot cues, loops) is copied to the new file. After conversion, the original files are moved to the SuperBox Archive Quarantine folder, so you can recover them if needed.

How long will this take?

Conversion speed depends on file count and your drive speed. MP3 encoding is typically fastest, FLAC slowest (uses compression level 8). A 10,000-track library might take several hours. The operation shows real-time progress.

Organize Library — Consolidate into Artist / Album / Track
Scans one or more folders (no matter how deeply nested) and moves or copies every track to the correct Artist / Album / Track location. Merges duplicate artist folders, creates missing folders, and prunes empty ones.
Moves Files
What structure does this create? 
Target Root/
├── Artist Name/
│   └── Album or Release Name/
│       └── track.mp3
├── Orphaned Tracks/       ← no artist tag
│   └── 2023/
│       └── track.mp3
└── Live Sets & Mixes/    ← duration ≥ mix threshold
    └── 2024/
        └── mix.mp3

Joint releases (e.g. "Artist A & Artist B") get their own folder under that combined name. Tracks without an album tag sit directly in the artist folder.

Is this safe to run?

Always run a Dry Run first. Dry Run scans everything and shows exactly where each file would go — nothing is moved. Review the output in the terminal log, then uncheck Dry Run and run again to execute.

Files already at their correct canonical location are skipped. If a filename clash is detected, the incoming file is renamed with a numeric suffix (_1, _2 …) rather than overwriting.

How does it handle artist name variations?

It reads the Album Artist tag (TPE2) first, which is typically the primary artist without "feat." credits. If that's missing it falls back to the Track Artist tag (TPE1). This means "Daft Punk feat. Pharrell Williams" lands in the Daft Punk folder, not a separate "feat." folder.

What counts as a Mix / Live Set?

Any track whose duration is at or above the Mix Threshold (default 15 minutes) is routed to Live Sets & Mixes / YYYY / instead of the normal Artist tree. Adjust the threshold if your library has long studio tracks you don't want caught.

Cross-Library Rescue — find unique tracks on other drives

Novelty Scanner — Capture Unique Tracks from Any Drive
Scans a source drive for tracks that don't exist anywhere in your home library and copies every one of them across. When in doubt, it copies. Use this to safely rescue one-of-a-kind recordings from backup drives before anything gets lost.
SAFE
How is this different from Find Duplicates?

Find Duplicates is cold-blooded — it only flags a track as a duplicate when it's highly confident. That strictness protects you from accidental deletion.

The Novelty Scanner flips the logic: a track is copied unless we're highly confident it's already in your library. One extra copy on disk costs nothing. A missing irreplaceable recording costs everything.

Uses your existing scan index — tracks already fingerprinted during a BPM/key run skip straight past Chromaprint, making the scan much faster on large libraries.

Dry Run vs Live Run

Dry Run (default) scans and reports what would be copied without touching anything. The source drive is never modified in either mode — this tool only copies, never moves.

Live Run copies novel tracks into your destination using the same Artist / Album / Track structure as the Library Organizer. Already-structured files keep their folder layout.

Welcome to SuperBox

        
A free, open-source rekordbox library toolkit by Guthrie Entertainment LLC

      

      
    


    


      

        
Why this exists

        
Rekordbox is an excellent performance tool. As a library manager, it has real limits. It doesn't write analysis results to your actual audio files — only its own database. BPM, key, and path data all live inside Rekordbox's private database, which means a library rebuild, a drive rename, or a corrupt database wipes all of it with no recovery path.

        
Path breakage is the other one. Move a folder, rename a drive, plug into a different USB port — Rekordbox loses every file it can't find and gives you no tools to fix it in bulk. SuperBox was built to solve exactly these problems: to give DJs with large, mature libraries a safety net that Rekordbox doesn't provide, and to ensure your library data outlives any single piece of software.

      


      

        
Core pipeline — each tool feeds the next

        
Run these in order on a fresh library. Each step's output is used as input by the steps that follow it.


        
▣ Library Audit — two passes in one run. The database pass reads your Rekordbox database and flags orphaned entries, broken file paths, untagged tracks, and duplicates the DB already knows about. The physical scan pass walks your entire music drive and maps every audio file on disk — extension breakdown, total size, trash-adjacent files, and a complete file inventory. The combined report is written to JSON and feeds the Relocate, Import, and Prune tools downstream. Run this first, every time.


        
🏷 Tag Tracks — analyzes each audio file's actual waveform using librosa and writes BPM and musical key directly into the file's tags (ID3 for MP3/AIFF, Vorbis comments for FLAC/OGG). Tags travel with the file permanently, independent of any database or software. Run this before finding duplicates so the pre-filter can use accurate BPM and key values.


        
🔎 Find Duplicates — acoustic fingerprinting across your whole library via Chromaprint. Finds files that are the same recording regardless of filename, bitrate, format, or trim length. Pre-filters by BPM, key, and duration before fingerprinting to keep things fast on large libraries. Produces a CSV report that the Prune tool reads directly.


        
✂ Prune Duplicates — loads the duplicate report and lets you select which files to remove from each group. Confirmation is deliberately multi-step, with each confirm button placed at a different screen corner to prevent muscle-memory clicking through a destructive action. Files go to Trash by default; permanent deletion is a separate explicit choice. Reads the CSV from Find Duplicates automatically if available.


        
📍 Fix Broken Paths — when a drive mounts under a new name, or files move to a new folder, every path stored in Rekordbox's database breaks silently. This tool takes an old path prefix and a new path prefix, then updates every matching entry in the database in one operation. Always creates a timestamped database backup before writing.


        
＋ Import — scans a folder and adds new audio files to Rekordbox's database. Skips files already present. Dry-run mode shows exactly what would be added before committing. A full database backup is created before any write.


        
🔗 Link Playlists — associates imported tracks to Rekordbox playlists based on the folder structure they live in. If your files are organized into named subfolders, this maps those folder names to playlist names automatically. Run after Import.

      


      

        
Optional & transformative

        
Run after the core pipeline is clean. These tools modify audio files directly.


        
📢 Normalize — measures the integrated loudness of each track using the EBU R128 standard (the same standard used in broadcast and streaming), and re-encodes any file that falls outside your target range. Every original is preserved as a .bak file until the new version is verified readable. Keeps energy consistent across your sets without touching gain in Rekordbox.


        
🔄 Convert Format — re-encodes audio files in a folder to a target format. Useful for converting a batch of downloads to your preferred format before importing. Originals are preserved.


        
🗂 Organize (De-fragment Library) — moves files into a consistent Artist / Album / Track folder hierarchy based on their embedded tags. Uses the Album Artist tag (TPE2) rather than the Track Artist tag (TPE1) to avoid an explosion of per-feature-artist folders. Camelot key prefixes in artist names are stripped before folder creation. Dry-run mode available. In assimilate mode, tracks from multiple source drives are consolidated into one target library root.

      


      

        
Cross-library rescue

        
★ Novelty Scanner — scans a source drive or folder (a backup drive, a USB stick, an old hard disk) and identifies audio files that are not acoustically present in your main library. These are genuinely new-to-you tracks, not duplicates under different names. Found tracks can be copied or moved to your primary library for import. Useful for recovering tracks from old drives you thought you'd lost.

Under the hood

        

          

            
librosa

            
Open-source audio analysis from the music information retrieval research community. Beat tracking on the actual waveform for BPM; Krumhansl-Schmuckler algorithm on chroma features for key detection. Runs entirely on your machine, no internet required.

          

          

            
chromaprint + pyacoustid

            
Acoustic fingerprinting — the same technology behind AcoustID and MusicBrainz. Generates a compact signature from audio content. Two files with different names, bitrates, and lengths — if they're the same recording, this finds them.

          

          

            
pyrekordbox

            
Reads and writes Rekordbox's SQLite database directly. This is what lets SuperBox import tracks, fix paths, and link playlists without Rekordbox needing to be open. Rekordbox must be closed before any write operation.

          

          

            
mutagen

            
Industry-standard Python library for reading and writing audio file tags. Handles ID3 (MP3/AIFF), Vorbis comments (FLAC), and more. Used by Beets, MusicBrainz Picard, and most serious music metadata tools.

          

          

            
pyloudnorm + soundfile

            
EBU R128 loudness measurement and audio I/O. The broadcast and streaming standard. Measures integrated loudness across the whole track — not peak, not RMS — and re-encodes only what falls outside your target range.

          

          

            
Flask + Waitress

            
Local web server — no cloud, no CDN, no external network calls. Everything runs on localhost:5001. The browser is just a UI skin over Python processes running on your own machine. Shut the server down with the ⏻ Quit button.

          

        

      


      

        
Honest comparison

        

          

            
vs Mixed In Key

            
MIK is excellent and arguably the most accurate BPM/key tool built specifically for DJs. $58 one-time, polished, years of refinement. SuperBox uses librosa — solid and research-grade — but MIK has the edge on complex or atonal tracks. If you use MIK, SuperBox respects it: it won't overwrite existing tags unless you explicitly ask.

          

          

            
vs Lexicon

            
Lexicon is the best cross-platform DJ library manager right now, ~$10/month. If you work across Serato, Traktor, and Rekordbox, Lexicon is probably the right tool. SuperBox is Rekordbox-specific, free, and focused on the safety and recovery problems Lexicon doesn't prioritize. They solve different problems.

          

          

            
vs Rekordbox's own analysis

            
Rekordbox's BPM analysis is fine. The difference is where the results live. Rekordbox writes to its database only — results don't survive a rebuild and don't work elsewhere. SuperBox writes to the file tags. Those tags live in the audio file forever, across any software and any database rebuild.

          

          

            
vs MusicBrainz Picard

            
Picard identifies recordings and corrects metadata — titles, artists, albums. SuperBox deliberately doesn't touch any of that. The scope is narrow by design: BPM, key, loudness, path integrity, and library safety. Different tool, different job.

          

        

      


      

        
The philosophy

        
Every write operation in SuperBox creates a timestamped backup before touching anything. Database changes, audio rewrites, path updates — all logged and reversible. Files are moved to Trash rather than permanently deleted unless you explicitly choose otherwise. The goal is that nothing SuperBox has ever done to your library should be unrecoverable.

        
Rekordbox must be closed before any write operation. SuperBox will warn you if it detects Rekordbox is running.

        
No cloud. No accounts. No subscription. No internet dependency at runtime. Your library data stays on your machine.
